Preface

In 2009, as we were wrapping up ES5, I was thinking about how to organize the material. I designed a hypertext encyclopedia format specifically for programming languages. I think the format would have worked well, but I never finished the project.

It is easy to forget that the web was originally intended to be a hypertext system. The JavaScript Encyclopedia would have used hypertext very effectively. Many of the articles would contain code that demonstrated or implemented the concept.

I intended that The JavaScript Encyclopedia be finished when ES5 was released. Not only did I miss that deadline, but since then ES6 has been published, which was itself four years late. So it is starting to become clear that I am not going to get this done on my own. I need help.

The prototype material can be seen at http://www.crockford.com/javascript/encyclopedia.

Cyc

The most important thing I learned in doing the prototype was that HTML is a horrible language for writing hypertext. It is clumsy, fussy, ugly, and error-prone. So I designed Cyc, a simple language with only one reserved character. By writing in Cyc, we can use Cyc processors that can produce

  • a single HTML file
  • an HTML file for each chapter
  • an HTML file for each specimen
  • an HTML file for each article
  • a TEX manuscript
  • a PostScript manuscript

A source is a set of enviroments that can contain other environments and content. The environments are

  • @book
  • @chapter
  • @appendix
  • @specimen
  • @article
  • @section
  • @program
  • @es5
  • @table
  • @list
  • @link
  • @url
  • @b
  • @i
  • @t
  • @reserved@
  • @aka
    •

    A book many contain one or more chapters. A chapter contains one or more specimens, which is can be a reserved word, name, concept, or special character. A specimen may contain one or more articles. An article many contain sections.

    Paragraphs are indicated with a blank line.

    A program is a JavaScript program that is illustrating or implementing a concept.

    Some paragraphs will be marked with an ES5 flag, indicating a new feature that is not availible with, or somehow incompatible with language processors that implement ECMAScript, Fifth Edition. With each new edition, we should remove all of the old flags, and then add new flags to highlight the latest incompatibilities.

    A table is a two dimensional grid structure. @| and @_ are used to separate the cells and rows. @! is used inside of a cell to indicate a header.

    A list gives each paragraph a bullet.

    A link refers to another feature of the hypertext. A url identifies a feature outside of the hypertext.

    Text can be marked as bold or italics or typewriter-like. Programmers like their programs to look like they were written with typewriters because they love antiques and steam.

    Special marks show specimens that are reserved words. Descriptive marks can be placed after special characters and operators to give them names.

    There are 6 sets of bracketing characters. They are all available all of the time.

  • ' '
  • ( )
  • [ ]
  • { }
  • < >

@specimen "% @aka (percent sign)"
@article "@t(%) @link <infix operator> @aka (remainder)"
The @t(%) @link <infix operator> computes the @link <remainder> produced by
the @link <division> of two @link<operand>s, a @link <dividend> and a
@link <divisor>. If the @link<operand>s are not numbers,
  it attempts to coerce them to numbers. If either @link <operand> cannot be coerced to a
  number or is @link <NaN>, or if the @link <dividend>
  is an @link <Infinity>, or
  if the @link <divisor> is a @link <zero>,
  then the result is @link <NaN>. Otherwise, if
  the @link <divisor> is an @link <Infinity>,
  the result is the @link <dividend>.

The @t(%) @link <operator produces> the same result as this function:

@begin(pgm)
function remainder(left, right) {
    var dividend = left,
        divisor = right,
        quotient = dividend / divisor;
    if (divisor === Infinity || divisor === -Infinity) {
        return isFinite(dividend) ? dividend : NaN;
    } else {
        return dividend - (divisor * Math[quotient >= 0
            ? 'floor'
            : 'ceil](quotient));
    }
}
@end(pgm)

The sign of the result will be the sign of the dividend, not
  the divisor. This differs from the @link <modulo> @link <infix operator> found in some other
  languages, where the result has the sign of the divisor. The modulo function
  can be computed with the following function, which adds the
  divisor to the remainder if the remainder has the wrong sign:

@begin(pgm)
function modulo(left, right) {
    var dividend = <a href="#plus prefix operator">+>left,
        divisor = <a href="#plus prefix operator">+>right,
        mod = dividend % divisor;
    if (mod && (divisor > 0) === (mod < 0)) {
        mod += divisor;
    }
    return mod;
}
@end(pgm)

@article "& @link <infix operator> @aka (bitwise and)"
The @t(&) infix operator performs the bitwise <var>and</var> operation. The <var>and</var> operation uses 32 bits from each @link<operand>. If two corresponding bits are 1, then the corresponding result bit is 1. Otherwise, the corresponding result bit is 0. The @link<operand>s are converted from a number (64 bit floating point) to a 32-bit <a href="I.html#integer">integer>. It produces the same result as this function:
@pgm {
function bitwise_and(left, right) {
    var result = 0, bit, i, left_int, right_int,
        two_32 = 4294967296,    // Math.pow(2, 32)
        two_31 = 2147483648;    // Math.pow(2, 31)

// Make 32-bit unsigned integers from the @link<operand>s. The % is a remainder
// operator, not a modulo operator, so some sign correction may be necessary.

    left_int = Math[left >= 0 ? 'floor' : 'ceil'](left) % two_32;
    if (left_int < 0) {
        left_int += two_32;
    }
    right_int = Math[right >= 0 ? 'floor' : 'ceil'](right) % two_32;
    if (right_int < 0) {
        right_int += two_32;
    }

// For each of the 32 bits,
//      Double the current result
//      Set the new bit to 1
//      If the most significant bits are set, then subtract them out
//      Otherwise clear the new bit
//      Add the new bit to the result
//      Double the ints, rotating the next most significant bit into position

    for (i = 0; i < 32; i += 1) {
        result += result;
        bit = 1;
        if (left_int >= two_31) {
            left_int -= two_31;
        } else {
            bit = 0;
        }
        if (right_int >= two_31) {
            right_int -= two_31;
        } else {
            bit = 0;
        }
        result += bit;
        left_int += left_int;
        right_int += right_int;
    }

// If the most significant bit of the result is set, then produce a negative.

    if (result >= two_31) {
        result -= two_32;
    }
    return result;
}}

@article "=== infix operator"><code>===) @link <infix operator> @aka (equal)"
The <code>===) operator compares two values, producing @link((true)> if they are
  equal and @link <false> if they are not. Two strings
  are considered equal if they have the same <a href="L.html#length">@t(length)> and contain exactly the same code points in the same sequence.

The expression@t( x === x )will always result in @link((true)> except when the value of@t( x )is @link <NaN>.
  Surprisingly, there exist mathematicians that think this was a good idea. It presents
  an obvious programming hazard. For example, suppose you need a function that tests that a function
  returns the correct value. Special hanling is required for the case where the expected value is @link <NaN><a href="N.">>. Simply using <code>===) to compare the expected value and the result will
  do the wrong thing when the expected value is @link <NaN>, so
  much more complicated comparisons are required using the <a href="T.html#typeof prefix operator">@t(typeof) prefix operator> and the <a href="I.html#isNaN global function">@t(isNaN) global function>.
  (The <a href="I.html#isNaN global function">@t(isNaN) global function> can produce incorrect results when its argument is a string.)

@article "| infix operator">@t(|) @link <infix operator> @aka (bitwise or)"
The @t(|) operator performs the bitwise <var>or</var> operation. The <var>or</var> operation uses 32 bits from each @link<operand>. If two corresponding bits are 0, then the corresponding result bit is 0. Otherwise, the corresponding result bit is 1. The @link<operand>s are converted from a number (64 bit floating point) to a 32-bit <a href="I.html#integer">integer>.
@pgm {
function bitwise_or(left, right) {
    var result = 0, bit, i, left_int, right_int,
        two_32 = Math.pow(2, 32),    // 4294967296
        two_31 = Math.pow(2, 31);    // 2147483648

// Make 32-bit unsigned integers from the @link<operand>s. The % is a remainder
// operator, not a modulo operator, so some sign correction may be necessary.

    left_int = Math[left >= 0 ? 'floor' : 'ceil'](left) % two_32;
    if (left_int < 0) {
        left_int += two_32;
    }
    right_int = Math[right >= 0 ? 'floor' : 'ceil'](right) % two_32;
    if (right_int < 0) {
        right_int += two_32;
    }

// For each of the 32 bits,
//      Double the current result
//      Set the new bit to 0
//      If the most significant bits are set, then subtract out the bits out
//          and set the new bit
//      Add the new bit to the result
//      Double the ints, rotating the next most significant bit into position

    for (i = 0; i < 32; i += 1) {
        result += result;
        bit = 0;
        if (left_int >= two_31) {
            left_int -= two_31;
            bit = 1;
        }
        if (right_int >= two_31) {
            right_int -= two_31;
            bit = 1;
        }
        result += bit;
        left_int += left_int;
        right_int += right_int;
    }

// If the most significant bit of the result is set, then produce a negative.

    if (result >= two_31) {
        result -= two_32;
    }
    return result;
}}
